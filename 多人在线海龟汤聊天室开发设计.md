# 多人在线海龟汤聊天室开发设计

> 文档状态：已合并房间与临时积分规则  
> 项目方向：从零开发多人在线海龟汤聊天室  
> 优先技术栈：Next.js App Router + Route Handlers（API）+ Supabase  
> 已有资产：仅有用于储存题目信息的 `questions.json`  
> 编写日期：2026-06-13  
> 房间规则依据：`多人项目聊天室房间设计.md`；如有冲突，以该文件为准

## 1. 项目目标

本项目将从零开发一个多人在线聊天室，并将海龟汤作为聊天室中的核心游戏玩法：

- 已登录且积分足够的用户可以创建房间；加入房间不要求登录。
- 多名玩家可以在同一个房间实时聊天。
- 房间可以进行普通聊天，也可以开启一局海龟汤游戏。
- 海龟汤游戏中的题目、提问、提示、推理和结算仍由系统管理。
- AI 可以担任主持人，也可以由真人房主担任主持人。
- 同一房间中的玩家共享汤面、问答记录、已知事实和游戏进度。
- 房主创建房间时预付每个座位的临时积分，玩家使用所在座位的临时积分进行提问、请求提示和提交推理。
- 房间关闭时，未使用的临时积分退还给房主。

可以把新系统理解为：

> 聊天室是“房间”，海龟汤游戏是可以在房间中启动的一种“活动”。

这样设计后，未来还可以在聊天室中加入投票、狼人杀、你画我猜等其他活动，而不需要再次推翻聊天室的基础结构。

## 2. 项目基础与建设原则

### 2.1 当前已有内容

默认开发者没有旧项目代码、数据库或可复用的服务。当前唯一已有资产为：

- `questions.json`：保存海龟汤题目信息。

`questions.json` 中预计包含或需要补充：

- 题目 ID。
- 题目标题。
- 汤面。
- 汤底。
- 难度。
- 关键事实点。
- 示例提问。
- 可选的分级提示和内容标签。

### 2.2 从零建设原则

- 以多人房间为系统中心，不先制作单人模式再改多人模式。
- 房间聊天与海龟汤游戏状态分开设计。
- `questions.json` 只负责保存题目信息，不负责保存房间或游戏进度。
- 汤底只保存在服务端，游戏结束前不能发送给普通玩家。
- 所有业务结果使用结构化字段，不通过解析显示文字判断。
- 第一版优先完成可稳定游玩的最小功能；房间临时积分属于核心防滥用机制，必须随第一版实现。
- 创建房间必须登录，加入房间可以使用游客身份；权限设计不能假设所有房间成员都有平台账号。
- 所有正式提问、提示和推理都必须由服务端原子扣减积分，不能只在前端显示余额。
- 实时消息、掉线重连和并发提问应从架构设计阶段考虑。

## 3. 用户与房间角色

### 3.1 平台身份

- **注册用户**：拥有平台账号和个人积分，可以创建房间，也可以加入房间和参加游戏。
- **游客**：不需要登录即可加入房间，使用房间座位的临时积分；游客不能创建房间，也没有可跨房间使用的个人积分。
- **平台管理员**：管理用户、题库、违规消息和异常房间。

平台管理员属于网站级权限，不等同于某个房间的房主。

### 3.2 房间角色

- **房主**：创建房间的已登录用户，负责房间设置、选择、开启、关闭或切换题目，以及踢出成员、修改密码和关闭房间。
- **主持人**：知道汤底并负责回答问题的人。可以是真人，也可以是 AI。
- **玩家**：不知道汤底，通过提问和推理还原真相。
- **观众**：只观看游戏，默认不能提交正式提问或推理。

第一版可以简化为“房主、玩家”两个角色，并默认由 AI 主持。

## 4. 房间模型

每个房间至少包含以下信息：

| 字段 | 说明 |
| --- | --- |
| `id` | 房间唯一编号 |
| `name` | 房间名称 |
| `owner_id` | 房主用户 ID |
| `status` | 房间状态 |
| `password_hash` | 可选的房间密码哈希；不得保存明文密码 |
| `max_members` | 房主创建时预设的座位数量，至少 1 个 |
| `points_per_seat` | 每个座位预存的临时积分，默认 15 |
| `reserved_points` | 创建房间时从房主个人积分中预付的总积分 |
| `remaining_room_points` | 房间全部座位当前剩余的临时积分总额 |
| `created_at` | 创建时间 |
| `updated_at` | 最后更新时间 |
| `closed_at` | 房间关闭时间 |

建议的房间状态：

- `waiting`：等待玩家，尚未开始游戏。
- `playing`：海龟汤游戏进行中。
- `closed`：房间已经关闭。

`finished` 应作为 `GameSession` 的状态，而不是房间状态。房间和游戏局次应当分开：一个房间可以连续玩多道题，每道题对应一条独立的 `GameSession`。

### 4.1 创建房间

创建房间必须满足：

1. 创建者已经登录并拥有平台账号。
2. 房主预先设置座位数量，座位数至少 1 个。
3. 房主设置每个座位的临时积分，默认每个座位 15 积分。
4. 房主个人积分不少于 `座位数量 × 每座位临时积分`。
5. 房主可以设置可选的进入密码。

创建成功时，系统必须在一个数据库事务中：

1. 从房主个人积分扣除预付总额。
2. 创建房间。
3. 创建固定数量的房间座位。
4. 为每个座位写入初始临时积分。
5. 记录一笔可审计的积分流水。

如果任何一步失败，整个事务回滚，不能出现“积分已扣但房间未创建”或“房间已创建但未扣积分”的情况。

### 4.2 加入房间与座位

- 加入房间不要求登录，注册用户和游客都可以进入。
- 玩家进入时占用一个空闲座位，并使用该座位当前的临时积分。
- 有密码的房间必须先由服务端校验密码。
- 游客进入时需设置房间内昵称，并获得不可猜测的访客会话令牌。
- 游客刷新或短暂掉线后，应凭访客会话令牌恢复原座位，不能自动领取一个新座位和一份新积分。
- 房主踢出玩家后释放该座位，但不清空该座位剩余临时积分；被踢玩家允许再次加入。
- 玩家主动退出后是否短暂保留座位，可设置一个重连宽限期；宽限期结束后释放座位。

### 4.3 临时积分与操作费用

临时积分属于房间座位，不属于占用座位的玩家个人账户，也不能带出房间。

| 操作 | 消耗 |
| --- | ---: |
| 普通聊天 | 0 积分 |
| 正式提问 | 1 积分 |
| 请求提示 | 1 积分 |
| 提交推理 | 2 积分 |

扣分规则：

- 服务端必须先检查座位余额，再接受正式操作。
- 检查余额、扣除积分、创建消息或 AI 任务必须在同一个事务中完成。
- 余额不足时拒绝操作，并返回当前余额和所需积分。
- AI 调用失败且未产生有效结果时，应退回本次扣除的临时积分，并记录退款流水。
- 房主可以把自己的个人积分赠送给任意玩家当前占用的座位；赠送后积分变为该座位的临时积分。
- 所有扣除、赠送、退款和关房返还都必须写入积分流水，防止重复扣款或重复退款。

### 4.4 题目与聊天记录

- 房主可以随时从题库中选择并开启题目，也可以关闭当前题目或切换到其他题目。
- 开启、关闭或切换题目不会清空房间聊天记录。
- 每次操作都要在聊天流中广播一条 `system` 消息。
- 切换题目时，应结束当前 `GameSession` 并创建新的 `GameSession`；事实、提示、问答和推理记录按局次隔离。
- 聊天记录属于房间并持续保留，游戏事实和进度属于具体局次。

### 4.5 关闭房间与退款

只有房主或平台管理员可以关闭房间。关闭时必须：

1. 拒绝新的加入、扣分和 AI 请求。
2. 处理或取消尚未完成的 AI 任务，并结清需要退回的操作积分。
3. 汇总所有座位未使用的临时积分。
4. 将未使用积分退还到房主个人积分账户。
5. 写入唯一的关房退款流水，并把房间状态改为 `closed`。

关闭流程必须具备幂等性：即使接口因网络问题被重复调用，也只能退款一次。房主赠送到座位但尚未使用的积分同样属于未使用临时积分，关闭房间时一并退还房主。

## 5. 聊天消息类型

多人聊天室不能只用“用户消息”和“AI 消息”区分内容。建议为消息增加明确的业务类型：

| 消息类型 | 用途 |
| --- | --- |
| `chat` | 普通聊天，不触发海龟汤逻辑 |
| `question` | 向主持人提出问题 |
| `hint_request` | 请求提示 |
| `reasoning` | 提交完整或阶段性推理 |
| `host_answer` | 主持人对问题的回答 |
| `hint` | 主持人给出的提示 |
| `reasoning_result` | 系统给出的推理判定 |
| `system` | 加入房间、退出房间、开始游戏等系统通知 |
| `points_changed` | 扣分、赠送、退款或余额变化通知 |

前端最好提供明确按钮：

- 发送聊天
- 提交问题
- 请求提示
- 提交推理

同时提供 AI 自动分类作为辅助，但不应完全依赖 AI 猜测玩家意图。明确按钮就像给不同类型的信件准备不同信封，可以明显降低“普通聊天被当成正式提问”的错误。

## 6. 海龟汤核心逻辑

海龟汤部分是本项目的核心业务能力。

### 6.1 输入分类

玩家发送与游戏相关的内容后，系统将内容分为三类：

1. **提问 `question`**
   - 玩家提出一个希望主持人判断的问题。
   - 通常应该能够使用“是”或“否”回答。
   - 例如：“男人以前是不是遇到过海难？”

2. **提示 `hint`**
   - 玩家主动请求系统给出线索。
   - 例如：“给一点提示”“我们卡住了”“提示一下方向”。

3. **推理 `reasoning`**
   - 玩家正在提交一段解释、猜测或完整答案。
   - 例如：“我认为男人以前被困在荒岛，吃到的其实是人肉。”

推荐的判断顺序：

```text
玩家明确点击消息类型按钮
        ↓
有明确类型时直接使用
        ↓
没有明确类型时，检查“提示”“推理”等关键词
        ↓
仍无法确定时，再调用 AI 分类
        ↓
AI 仍无法确定时，默认按普通聊天或普通提问处理
        ↓
类型确定后，服务端按操作费用检查并扣除座位临时积分
```

### 6.2 普通提问的四种回答

普通提问必须返回以下四种回答之一：

#### 是 `yes`

问题描述的内容得到汤底明确支持。

示例：

```text
问题：男人以前是不是遇到过海难？
回答：是
```

#### 否 `no`

问题描述的内容与汤底明确冲突。

示例：

```text
问题：男人是因为食物中毒而自杀的吗？
回答：否
```

#### 与此无关 `irrelevant`

问题涉及的信息与真相没有关系，汤底也没有必要依赖该信息。

示例：

```text
问题：餐厅的老板是不是男人的亲戚？
回答：与此无关
```

“汤底没有写”不一定都等于“与此无关”。系统还要判断这个信息是否可能影响核心真相，避免过早排除合理方向。

#### 模糊问题 `ambiguous`

问题无法可靠地用“是”或“否”判断，或者问题本身存在多个解释。

示例：

```text
问题：海龟汤好喝吗？
回答：模糊问题
```

前端可以为四种回答设置固定颜色：

- 是：绿色。
- 否：红色。
- 与此无关：黄色。
- 模糊问题：蓝色或灰色。

### 6.3 回答约束

AI 主持人回答普通提问时必须遵守：

- 只能从汤底和题库提供的信息进行判断。
- 不得补充题库中不存在的新设定。
- 不得为了让故事“更合理”而自行编剧情。
- 不直接泄露完整汤底。
- 优先参考题库中的示例提问。
- 输出必须使用结构化数据，不能只依靠自然语言解析。

推荐输出格式：

```json
{
  "answer_type": "yes",
  "summary": "男人曾遭遇海难，被困在荒岛。",
  "reason": "汤底明确说明男人是一名海难幸存者。"
}
```

其中 `reason` 只用于系统日志、真人主持人辅助或管理员调试，默认不展示给普通玩家。

## 7. 事实总结

### 7.1 事实总结的作用

聊天消息会越来越多，如果玩家必须不断向上翻找，就容易丢失线索。事实总结相当于房间中的“公共白板”，记录已经确认的信息。

例如：

```text
男人曾遭遇海难。
男人在餐厅喝到的是真正的海龟汤。
男人不是因为食物中毒而自杀。
```

### 7.2 什么时候生成事实

- 回答为“是”时，通常生成一条肯定事实。
- 回答为“否”时，可以生成一条否定事实。
- 回答为“与此无关”时，不生成事实。
- 回答为“模糊问题”时，不生成事实。
- 系统给出提示时，可以将提示中公开的信息加入事实。
- 推理判定通常不直接加入事实，除非系统明确公开了某个事实点。

### 7.3 多人模式下的事实规则

- 事实属于当前游戏局次，不属于某一个玩家。
- 所有玩家看到同一份事实列表。
- 相同或近似事实应当合并，避免重复刷屏。
- 每条事实记录来源消息和发现者。
- 主持人可以撤销错误事实。
- 游戏结束后，事实列表应当保留在本局回放中。

建议的数据结构：

```json
{
  "id": 101,
  "game_session_id": 20,
  "content": "男人曾遭遇海难，被困在荒岛。",
  "polarity": "positive",
  "source_message_id": 880,
  "discovered_by": 12,
  "created_at": "2026-06-13T20:00:00+08:00"
}
```

## 8. 提示逻辑

提示应当帮助玩家改变方向，但不能直接公布汤底。

生成提示时需要向 AI 提供：

- 汤面。
- 汤底。
- 房间已经发现的事实。
- 最近若干条有效提问和回答。
- 玩家本次请求提示时附带的问题。
- 已经给过的提示，防止重复。

提示可以分为三个等级：

1. **轻提示**
   - 只提示关注方向。
   - 示例：“可以想一想男人以前是否喝过类似的汤。”

2. **中提示**
   - 公开一个边缘事实。
   - 示例：“男人过去经历过一次严重的生存危机。”

3. **强提示**
   - 公开接近关键事实的线索，但仍不直接说出完整答案。
   - 示例：“当年有人故意隐瞒了男人吃下的食物是什么。”

第一版可以由系统根据请求次数逐步提高提示等级，也可以由房主设置允许的最大提示等级。

## 9. 推理判定

### 9.1 关键事实点

每道题需要把完整汤底拆成若干个关键事实点。玩家的推理不需要逐字一致，只要表达出相同核心含义，就算覆盖该事实点。

示例：

```json
{
  "id": 1,
  "text": "男人曾遭遇海难，被困荒岛",
  "accept": ["海难", "荒岛", "船难", "遇难", "漂流"]
}
```

### 9.2 判定结果

建议第一版按关键事实点覆盖比例判定：

- 覆盖率大于等于 70%：`correct`，推理正确。
- 覆盖率大于等于 20% 且小于 70%：`partially_correct`，部分正确。
- 覆盖率小于 20%：`wrong`，推理错误。

建议将比例设置为配置项：

```json
{
  "correct_threshold": 0.7,
  "partial_threshold": 0.2
}
```

### 9.3 多人模式下的推理流程

1. 某个玩家提交推理。
2. 系统将该消息标记为 `reasoning`。
3. AI 对每个关键事实点进行覆盖判断。
4. 系统计算覆盖率。
5. 将结果广播给房间内所有用户。
6. 如果推理正确，结束本局并公开汤底。
7. 如果部分正确，只提示“部分正确”，默认不公开具体缺失点。
8. 如果推理错误，游戏继续。

可选规则：

- 只有房主可以发起最终答案投票。
- 某个玩家提交推理后，其他玩家投票决定是否作为团队最终答案。
- 房间可以设置个人胜利或团队胜利。

## 10. 游戏完整流程

```text
房主创建房间
    ↓
系统从房主账户预扣各座位临时积分
    ↓
其他玩家以注册用户或游客身份加入并占用座位
    ↓
房主选择题目和主持方式
    ↓
系统创建 GameSession
    ↓
向所有玩家公开汤面
    ↓
玩家聊天，或使用座位临时积分提问、请求提示和提交推理
    ↓
主持人或 AI 回答
    ↓
系统更新问答记录和公共事实
    ↓
玩家提交正确推理，或房主主动结束
    ↓
公开汤底、关键事实点和本局统计
    ↓
房间返回等待状态，可继续下一题
    ↓
房主关闭房间时，未使用临时积分退还房主
```

## 11. `questions.json` 题库样式

### 11.1 题目基础字段

每道题建议包含：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 题目唯一编号 |
| `title` | 是 | 题目标题 |

| `points` | 是 | 推理判定使用的关键事实点 |
| `examples` | 建议 | 示例提问及标准回答 |
| `tags` | 否 | 题目标签 |
| `hint_levels` | 否 | 人工编写的分级提示 |
| `answer_config` | 否 | 推理评分阈值等设置 |
| `content_warning` | 否 | 死亡、暴力等内容提示 |
| `version` | 建议 | 题目版本 |

### 11.3 题库编写要求

- 汤面应当简短、有悬念，但不能故意使用无法解释的错误信息。
- 汤底应当完整解释汤面中的异常现象。
- 每个关键事实点只表达一个核心事实。
- `accept` 只用于辅助匹配，不能代替 AI 语义判断。
- 示例提问需要覆盖四种回答。
- 示例中的回答必须与汤底一致。
- 示例总结应当能直接放入事实列表。
- 不确定或无关的示例不应生成事实总结。
- 修改汤底后，必须同步检查关键事实点、提示和示例提问。
- `questions.json` 应当只保存合法 JSON 数据，不能包含 JavaScript 变量、注释、函数或尾随逗号。
- 服务启动时应校验题目必填字段，避免游戏开始后才发现数据缺失。
- 服务端读取题目后，对外只返回允许公开的字段，不得直接返回完整题目对象。

### 11.4 题库公开与保密字段

游戏开始时可以向玩家公开：

```javascript
{
  id: 1,
  title: "餐厅自杀",
  surface: "一个男人走进一家餐厅……",
  difficulty: "中等",
  tags: ["经典", "海难", "黑暗"],
  content_warning: ["死亡", "食人"]
}
```

以下字段只能由服务端、AI 主持人或真人主持人读取：

- `bottom`
- `points`
- `examples` 中的标准回答、原因和总结
- `hint_levels` 中尚未公开的提示
- `answer_config`

不能把完整 `questions.json` 放进 Next.js 的 `public/` 目录，也不能下载到浏览器后再隐藏汤底。这样就像把答案放在试卷背面，虽然界面没有显示，但玩家仍然可以通过开发者工具查看。

## 12. 推荐数据模型

### 12.1 核心表

建议将数据库拆分为以下主要实体：

- `users`：用户。
- `rooms`：聊天室。
- `room_seats`：房间预设座位、当前占用者和临时积分余额。
- `room_members`：房间成员、游客会话、昵称、角色及所占座位。
- `room_messages`：所有聊天室消息。
- `game_sessions`：某个房间中的一局游戏，记录 `question_id` 和 `question_version`。
- `game_facts`：当前局已经公开的事实。
- `reasoning_attempts`：玩家提交的推理与判定结果。
- `game_hints`：已经公开的提示。
- `point_ledger`：个人积分和房间临时积分的扣除、赠送、退款与关房返还流水。

第一版不需要在 Supabase 中建立 `questions` 表，题库的唯一来源是服务端 `questions.json`。如果以后需要网页管理题库、在线投稿、审核或不发版更新题目，再将题库迁移到数据库。

### 12.2 关系说明

```text
User ──< Room
          │
          ├──< RoomSeat ──0..1── RoomMember
          ├──< RoomMessage
          ├──< PointLedger
          └──< GameSession
                   │
                   ├──< GameFact
                   ├──< GameHint
                   └──< ReasoningAttempt
```

`GameSession.question_id` 对应 `questions.json` 中的题目 ID，但它不是数据库外键。Route Handler 必须检查该 ID 是否真实存在。

`RoomMember.user_id` 对注册用户有值，对游客可以为空；游客必须使用服务端签发并安全保存摘要的 `guest_token` 识别。`RoomSeat` 的积分余额应使用数据库约束保证不小于 0。

## 13. 实时通信

多人聊天室需要服务器主动把新消息推送给所有玩家。普通 HTTP 请求像“玩家不断去前台问有没有新消息”，实时连接则像“前台有消息就直接广播”。

第一版优先使用：

- **Next.js App Router + TypeScript**：页面、服务端组件和前端交互。
- **Next.js Route Handlers**：放在 `app/api/**/route.ts` 中，承担创建房间、开始游戏、提交问题和调用 AI 等服务端接口。本文后续将其统称为 API Routes。
- **Supabase Auth**：注册用户的注册、登录、退出和用户会话。
- **Supabase PostgreSQL**：保存房间、成员、消息、游戏局次、事实和推理记录。
- **Supabase Realtime**：订阅房间消息、成员状态和游戏状态变化。
- **Supabase Row Level Security（RLS）**：限制注册用户只能读取和操作自己有权限访问的数据。
- **Vercel**：优先作为 Next.js 部署平台；Supabase 作为独立的后端服务。

不建议为了聊天室再建立 Express、NestJS 或常驻 Socket.IO 服务器。Next.js API 负责业务操作，Supabase Realtime 负责实时广播，可以减少第一版的部署和维护成本。

职责可以简单理解为：

```text
Next.js 页面              负责显示房间和游戏界面
Next.js Route Handlers    负责需要保密或校验的业务操作
Supabase Auth             负责确认“你是谁”
Supabase PostgreSQL       负责永久保存数据
Supabase Realtime         负责把数据变化及时告诉房间成员
Supabase RLS              负责确认“你能看什么、能改什么”
```

`questions.json` 应放在服务端可读取、浏览器不能直接访问的位置，例如项目根目录下的 `data/questions.json`。Route Handler 读取题目后，只返回汤面等公开字段。

游客没有 Supabase 登录身份，不应直接获得可任意查询数据库的匿名权限。推荐由 Route Handler 校验房间访客令牌后代为执行加入、发消息和游戏操作；实时订阅可以使用只暴露当前房间数据的受控频道或服务端签发的短期访问凭证。

关键实时事件或数据变化可以包括：

```text
room:join
room:leave
room:kicked
room:message
room:member_updated
room:password_changed
room:closed
room:points_changed
game:start
game:stop
game:switch
game:question
game:answer
game:hint
game:reasoning_result
game:fact_added
game:end
```

服务器必须验证请求者是房主、注册成员或持有有效访客令牌的游客，不能仅相信前端传来的 `room_id`、`member_id` 或 `seat_id`。

## 14. AI 主持人的并发处理

多人可能在几秒内连续提问。如果所有问题同时发送给 AI，回答可能乱序，事实列表也可能互相覆盖。

建议每个游戏房间建立一条 AI 处理队列：

1. 玩家问题进入等待队列。
2. 系统按提交时间逐条处理。
3. AI 回答完成后再广播。
4. 更新公共事实。
5. 继续处理下一条问题。

前端可以显示：

```text
问题已提交，前面还有 2 个问题等待主持人回答。
```

第一版也可以限制“同一时间只允许一个待回答的正式问题”，减少实现复杂度。

- 创建房间。

## 16. 权限和安全规则

- 未加入房间的人不能读取房间消息；游客也必须持有与成员记录匹配的有效访客令牌。
- 普通玩家不能获取汤底接口。
- 汤底不能随着汤面一起发送到前端再用 CSS 隐藏。
- 所有房主操作必须由服务器再次检查权限。
- 所有实时事件和写操作都要验证房主登录身份，或验证注册成员身份、游客会话和房间成员资格。
- 房间密码只能保存安全哈希，不得写入日志或返回前端。
- 创建房间、操作扣分、积分赠送、AI 失败退款和关房返还必须使用数据库事务与幂等键。
- 客户端传入的积分余额、扣分金额、座位归属和退款金额都不可信，必须由服务端重新计算。
- 限制发送频率，防止刷屏和重复触发 AI。
- 限制消息长度。
- 对用户昵称、房间名和消息进行 HTML 转义，防止脚本注入。
- AI 请求失败时保留原问题，并允许重试。
- 游戏结束后才允许普通玩家读取汤底。

## 17. 从零实现的核心服务

除 `questions.json` 外，以下模块均需要新建。模块名称可以根据最终目录结构调整。

### 17.1 用户服务

负责：

- 注册、登录和退出。
- 用户资料。
- 用户个人积分余额和积分流水。
- 登录状态验证。
- 平台管理员权限。

### 17.2 房间服务

负责：

- 创建和关闭房间。
- 加入和退出房间。
- 固定座位、游客会话、成员与角色。
- 房间密码校验与修改。
- 踢出成员并允许其再次加入。
- 创建房间时预付临时积分。
- 房主向指定座位赠送个人积分。
- 关闭房间时退还未使用临时积分。

### 17.3 实时消息服务

负责：

- 连接管理。
- 加入实时频道。
- 广播聊天消息。
- 在线状态。
- 断线重连和历史消息补发。

### 17.4 海龟汤游戏服务

负责：

- 从服务端的 `questions.json` 安全读取题目。
- 创建游戏局次。
- 公开汤面。
- 管理游戏状态。
- 保存问答、提示和公共事实。
- 结束游戏并公开汤底。
- 房主关闭或切换题目时结束当前局次，但保留房间聊天记录。

### 17.5 AI 主持服务

负责：

- 判断消息属于提问、提示还是推理。
- 对普通提问给出四种标准回答之一。
- 生成事实总结。
- 根据游戏进度生成提示。
- 根据关键事实点判断推理结果。
- 验证 AI 返回的数据格式。

建议对外统一返回结构化数据，例如：

```json
{
  "message_type": "host_answer",
  "answer_type": "yes",
  "is_game_ending": false
}
```

显示文字只是外观，结构化字段才是系统真正使用的结果。前端显示“是”或更换颜色时，不应改变服务端的业务判断。

## 18. 第一版 MVP 范围

第一版建议只完成最核心的多人玩法：

- 用户注册、登录和个人积分余额。
- 已登录用户创建房间，并预付固定座位的临时积分。
- 注册用户或游客加入和退出房间。
- 可选房间密码。
- 房间成员列表。
- 座位临时积分余额。
- 多人实时文字聊天。
- 房主选择、开启、关闭和切换题目，且不清空聊天记录。
- 房主踢出玩家、修改密码和向玩家座位赠送积分。
- AI 担任主持人。
- 玩家明确选择聊天、提问、提示或推理。
- 正式提问和提示消耗 1 积分，推理消耗 2 积分。
- 四种提问回答。
- 共享问答记录。
- 共享事实总结。
- 推理正确后公开汤底并结束游戏。
- 同一房间可以开始下一局。
- 房主关闭房间时，未使用临时积分只退还一次。

第一版暂不实现：

- 真人主持模式。
- 语音聊天。
- 好友和私聊。
- 积分商城、充值支付和复杂积分经济系统；但房间预付、扣分、赠送和退款属于第一版。
- 排行榜和赛季。
- 观众席。
- 多人投票判定。
- 自定义题目审核社区。

## 19. 推荐开发顺序

1. 确认 `questions.json` 的字段格式并编写题库校验器。
2. 使用 Next.js App Router 和 TypeScript 创建新项目。
3. 创建 Supabase 项目并配置本地与线上环境变量。
4. 设计用户积分、房间、固定座位、成员、消息、积分流水和游戏局次等 PostgreSQL 数据表。
5. 为注册用户可访问的表启用并编写 RLS 策略，并设计游客通过 Route Handler 和访客令牌访问的边界。
6. 接入 Supabase Auth，完成注册、登录、用户身份验证和个人积分余额。
7. 实现创建房间事务：预设至少 1 个座位、预付每座位临时积分和可选密码。
8. 实现注册用户或游客加入房间、占用和恢复座位。
9. 完成房主踢人、修改密码、赠送积分、关闭房间及幂等退款。
10. 接入 Supabase Realtime，实现普通多人聊天、成员和积分余额同步。
11. 增加 `GameSession`，让房主可以开启、关闭和切换题目且保留聊天记录。
12. 由 Route Handler 安全读取 `questions.json`，只返回汤面等允许字段。
13. 接入原子扣分、普通提问及四种回答，并在 AI 失败时退款。
14. 接入事实总结和提示。
15. 接入推理扣分、评分与游戏结算。
16. 增加并发控制、频率限制和异常恢复。
17. 完成游客重连、多人并发、积分守恒、重复退款、RLS 和 API 权限测试。

## 20. 验收标准

达到以下条件时，可以认为多人聊天室第一版完成：

- 一个已登录且积分足够的用户能创建房间，预付积分计算正确。
- 未登录游客能使用昵称和正确密码加入房间并占用一个座位。
- 注册用户和游客能同时进入同一房间。
- 任意一方发送聊天消息，另一方无需刷新即可看到。
- 房主能选择、开启、关闭和切换海龟汤题目，聊天记录不会被清空，并能看到对应系统提示。
- 所有玩家看到相同汤面，但不能提前获取汤底。
- 玩家提交普通提问或请求提示时扣除 1 积分，提交推理时扣除 2 积分。
- 余额不足时操作被拒绝，且不会调用 AI。
- 玩家能提交普通提问并得到四种标准回答之一；AI 失败时本次积分被退回。
- 所有玩家看到相同的问答记录和事实总结。
- 请求提示后，提示会同步给全房间。
- 提交推理后，系统能返回错误、部分正确或正确。
- 推理正确时，全房间同时看到汤底和结束状态。
- 玩家刷新或短暂掉线后，能恢复房间和本局记录。
- 游客刷新或短暂掉线后恢复原座位和余额，不会重复领取临时积分。
- 非房间成员不能读取房间消息。
- 普通玩家不能调用房主操作。
- 房主能踢出玩家、修改密码和向指定玩家座位赠送积分，被踢玩家可以再次加入。
- 房主关闭房间后，所有未使用临时积分退还至房主账户；重复关闭请求不会重复退款。
- 并发提问、赠送和关闭房间时，系统总积分保持守恒，座位余额不会小于 0。

## 21. 总结

本项目应当从一开始就以“聊天室房间”为系统中心，而不是先建立单人问答页面，再在外层补充多人消息。

新的结构中：

- 房间负责组织多人。
- 固定座位负责承载玩家在房间内可用的临时积分。
- 消息负责承载实时交流。
- 游戏局次负责海龟汤状态。
- 题库负责提供汤面、汤底、关键事实和示例。
- AI 主持服务负责分类、回答、提示和推理判定。
- 事实总结负责保存全体玩家已经确认的公共线索。
- 积分流水负责保证创建预付、操作扣分、房主赠送、失败退款和关房返还有据可查且只执行一次。

唯一已有的内容是 `questions.json` 中的题目资产。其余用户、游客会话、个人积分、房间座位、临时积分、实时通信、游戏状态、AI 主持和数据存储能力都应使用 Next.js 与 Supabase 按照多人在线场景重新建设，并为后续增加更多多人互动玩法留下空间。
