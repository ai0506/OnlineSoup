# OnlineSoup 本地部署教程（Windows）

这份教程适合第一次部署本项目的人。完成后：

- OnlineSoup 网页运行在你的电脑上；
- 数据库、注册登录和实时消息由 Supabase 云端项目提供；
- 浏览器访问地址为 `http://localhost:3000`。

可以把它理解成：网页这家“店”开在你的电脑上，Supabase 是远程的
“仓库和门禁系统”。这是本仓库目前最简单、最稳定的本地运行方式，不需要 Docker。

## 一、准备软件

### 1. 安装 Node.js

本项目使用 Next.js 16，需要 **Node.js 20.9.0 或更高版本**。

推荐安装 Node.js 的 LTS 版本。安装后打开 PowerShell，执行：

```powershell
node --version
npm.cmd --version
```

只要 Node.js 版本不低于 `20.9.0` 即可。

> 为什么教程写 `npm.cmd` 而不是 `npm`？
>
> 部分 Windows PowerShell 会拦截 `npm.ps1`，并提示“禁止运行脚本”。
> `npm.cmd` 做的是同一件事，但不会触发这个问题。如果你的 `npm` 能正常使用，
> 也可以把下文的 `npm.cmd` 全部换成 `npm`。

### 2. 注册 Supabase

打开 [Supabase](https://supabase.com/) 并注册账号。

## 二、安装项目依赖

在 PowerShell 中进入项目目录：

```powershell
cd D:\asw\Desktop\Create\PyCharm\OnlineSoup
```

根据仓库锁定的版本安装依赖：

```powershell
npm.cmd ci
```

如果 `npm ci` 因为你修改过依赖文件而失败，可以改用：

```powershell
npm.cmd install
```

安装结束后，项目目录中会出现 `node_modules` 文件夹。

## 三、创建 Supabase 项目

1. 登录 Supabase Dashboard。
2. 点击 **New project**。
3. 填写项目名称、数据库密码和区域。
4. 数据库密码请保存好；本项目日常启动不需要填写它，但以后管理数据库会用到。
5. 等待项目创建完成。

## 四、初始化数据库

`supabase/migrations` 目录下的迁移文件数量较多（且会持续增加），不建议像早期那样
逐个手动打开 SQL Editor 粘贴执行。推荐使用 Supabase CLI 一次性按文件名顺序部署：

```powershell
npx.cmd supabase login
npx.cmd supabase link --project-ref 你的项目编号
npx.cmd supabase db push --linked --dry-run
npx.cmd supabase db push --linked
```

- `--project-ref` 是 Supabase 项目 URL 中 `https://<project-ref>.supabase.co` 的那一段。
- `db push --linked --dry-run` 用来预览将要执行哪些迁移，确认无误后再去掉
  `--dry-run` 正式执行。
- 命令执行完成后应看到每个迁移文件的 `Applying migration ...`，最后输出
  `Finished supabase db push.`。

如果你只是想临时在 SQL Editor 里手动执行（例如排查某条具体迁移），仍然必须
**按文件名顺序**逐个执行 `supabase/migrations` 下的全部 `.sql` 文件，不能跳着执行，
也不能从中间某个文件开始。

如果连接 Supabase 数据库直连地址时出现 `tls error (EOF)`，改用 Session Pooler
方式部署，具体步骤见 `CLAUDE.md` / `AGENTS.md` 的"数据库迁移"章节。

执行完成后，可以在 **Table Editor** 中看到 `profiles`、`rooms`、`room_seats`、
`room_messages`、`puzzles`、`puzzle_progress`、`room_ai_requests` 等表。

> 如果某个迁移执行失败，不要继续执行后面的迁移。先确认是否漏掉了前一个文件，
> 或者是否在同一个 Supabase 项目中重复执行了整套迁移。

## 五、配置注册回调地址

在 Supabase Dashboard 中：

1. 打开 **Authentication**。
2. 进入 **URL Configuration**。
3. 将 **Site URL** 设置为：

```text
http://localhost:3000
```

4. 在 **Redirect URLs** 中添加：

```text
http://localhost:3000/auth/callback
```

地址必须完全一致。请统一使用 `localhost`，不要一会儿使用 `localhost`，
一会儿使用 `127.0.0.1`，否则邮箱验证后可能无法正确返回项目。

如果你只是自己测试，可以在 Authentication 的邮件设置中关闭邮箱确认。
如果保持邮箱确认开启，注册后需要点击 Supabase 发来的验证邮件。

## 六、填写环境变量

### 1. 获取 Supabase 地址和密钥

在 Supabase 项目顶部点击 **Connect**，或者进入 **Settings > API Keys**，
找到以下内容：

- Project URL；
- Publishable key，通常以 `sb_publishable_` 开头；
- Secret key，通常以 `sb_secret_` 开头。

Publishable key 像“前台访客证”，可以供网页使用，实际数据权限仍由 RLS 控制。
Secret key 像“总钥匙”，只能放在服务端，绝对不能发给别人或放进公开仓库。

### 2. 创建 `.env.local`

项目已经提供 `.env.example`。如果没有 `.env.local`，执行：

```powershell
Copy-Item .env.example .env.local
```

打开 `.env.local`，填写真实值：

```env
NEXT_PUBLIC_SUPABASE_URL=https://你的项目编号.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=sb_publishable_你的公开密钥
NEXT_PUBLIC_SITE_URL=http://localhost:3000
SUPABASE_SECRET_KEY=sb_secret_你的服务端密钥
ADMIN_EMAILS=你的邮箱@example.com
DEEPSEEK_API_KEY=sk-你的DeepSeek密钥
DEEPSEEK_MODEL=deepseek-v4-flash
```

注意：

- 不要给 `SUPABASE_SECRET_KEY` 添加 `NEXT_PUBLIC_` 前缀；
- 多个管理员邮箱使用英文逗号分隔；
- `.env.local` 已被 `.gitignore` 忽略，不要手动提交它；
- 修改环境变量后，需要停止并重新启动项目。
- `DEEPSEEK_API_KEY` / `DEEPSEEK_MODEL` 用于房间内的 AI 问答（询问、提示、尝试推理）。
  没有 DeepSeek Key 也可以正常使用注册登录、房间、座位、聊天等其他功能，只是房主
  开题后玩家无法使用询问/提示/推理这三个 AI 模式。

如果不需要后台管理功能，`ADMIN_EMAILS` 仍可填写你自己的测试邮箱。

## 七、启动项目

开发模式启动：

```powershell
npm.cmd run dev
```

看到类似下面的信息，就表示启动成功：

```text
Local: http://localhost:3000
```

然后在浏览器打开：

[http://localhost:3000](http://localhost:3000)

停止项目时，在 PowerShell 窗口按 `Ctrl + C`。

## 八、首次功能检查

建议按下面的顺序测试：

1. 打开 `/login` 注册账号。
2. 如果开启了邮箱确认，进入邮箱点击验证链接。
3. 登录后确认首页显示初始积分。
4. 创建一个房间。
5. 使用无痕窗口访问首页，以游客身份输入房间码并加入。
6. 在两个窗口中发送消息，确认聊天内容能够实时出现。
7. 房主在房间页的"海龟汤"标签中选择一道题目，确认玩家侧能看到题面，并能使用
   询问 / 提示 / 尝试推理（需要已配置 `DEEPSEEK_API_KEY`）。
8. 用 `ADMIN_EMAILS` 中的账号登录，打开 `/admin` 检查账户管理和题库管理页面。

## 十、常见问题

### PowerShell 提示“禁止运行脚本”

使用：

```powershell
npm.cmd run dev
```

不要使用 `npm run dev`。这不需要修改系统执行策略。

### 页面提示“缺少 Supabase 环境变量”

检查：

- 文件名是否确实是 `.env.local`，而不是 `.env.local.txt`；
- 五个变量名是否拼写正确；
- 等号右边是否已经替换为真实值；
- 修改文件后是否重启了开发服务器。

### 注册成功，但没有收到邮件

检查垃圾邮件箱，并在 Supabase 的 **Authentication > Logs** 查看发送记录。
个人测试时也可以暂时关闭邮箱确认。

### 点击验证邮件后跳转失败

确认 Supabase 的 Redirect URLs 中存在完整地址：

```text
http://localhost:3000/auth/callback
```

同时确认 `.env.local` 中：

```env
NEXT_PUBLIC_SITE_URL=http://localhost:3000
```

### 数据库报“表不存在”或“函数不存在”

通常是迁移文件没有执行完整。回到第四节，按文件名顺序检查每个 SQL 文件。

### 请求数据库时出现权限或 Data API 错误

确认 Supabase 项目的 **Data API** 设置中暴露了 `public` schema。
本项目的迁移已经为所需表、函数配置了授权和 RLS；Data API 像大楼入口，
RLS 像每个房间的门锁，两层都需要正常配置。

### 3000 端口被占用

可以关闭占用 3000 端口的程序，或者临时改用 3001：

```powershell
npm.cmd run dev -- -p 3001
```

如果使用 3001，还要同步修改 `.env.local` 和 Supabase URL Configuration：

```text
http://localhost:3001
http://localhost:3001/auth/callback
```
